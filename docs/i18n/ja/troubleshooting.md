# トラブルシューティング

翻訳実行が予期せず成功した場合、設定中に失敗した場合、またはレビューが必要な出力が生成された場合にこのページを使用してください。

## はじめに

1. まずは `translate -l "ko" -md` のような対象を絞ったコマンドを実行してください。
2. コンソールのデバッグログを有効にするには `-d` を追加します。
3. デバッグログを `<root-dir>/logs/` に保存するには `-s` を追加します。
4. 翻訳後に `co-op-review` を実行して、鮮度、構造、およびローカルリンクを確認してください。

```bash
translate -l "ko" -md -d -s
co-op-review -l "ko"
```

## 設定エラー

### 言語モデルプロバイダーがありません

エラー:

```text
No language model configuration found.
```

修正:

- Azure OpenAI または OpenAI を構成してください.
- コマンドが実行される環境に変数が存在することを確認してください.
- ローカルで使用する場合は、プロジェクトルートの `.env` に配置してください.

詳細は [設定](configuration.md) を参照してください。

### Azure AI Vision を利用しない画像翻訳

エラー:

```text
Image translation requested but Azure AI Service is not configured.
```

修正:

- `AZURE_AI_SERVICE_API_KEY` を追加してください.
- `AZURE_AI_SERVICE_ENDPOINT` を追加してください.
- または `translate -l "ko" -md` のようなテキストのみのコマンドを実行してください.

### キーまたはエンドポイントが無効

症状には `401`、権限エラーの省略表示、またはエンドポイントへのアクセスエラーが含まれることがあります。

修正:

- キーがエンドポイントと同じ Azure リソースに属していることを確認してください.
- `-img` を使用する場合、リソースが Vision をサポートしていることを確認してください.
- Azure OpenAI のデプロイ名と API バージョンがデプロイと一致していることを確認してください.
- デバッグログ付きで実行してください: `translate -l "ko" -md -d -s`.

## ファイルが翻訳されませんでした

主な原因:

- 選択したフラグがファイルと一致していない。
- 既に翻訳済みのファイルが存在している。
- ソースファイルが除外されたディレクトリの下にある。
- コマンドが誤ったプロジェクトルートから実行されている。

確認事項:

```bash
translate -l "ko" -md --dry-run
translate -l "ko" -nb --dry-run
translate -l "ko" -img --dry-run
```

コマンドがプロジェクトルート外で実行される場合は `--root-dir` を使用してください。

## 予期しないリンクの挙動

リンクの書き換えは選択されたコンテンツタイプに依存します:

- `-nb` を含めた場合: ノートブックのリンクは翻訳済みノートブックを指すことがあります。
- `-nb` を除外した場合: ノートブックのリンクはソースのノートブックを指したままになることがあります。
- `-img` を含めた場合: 画像リンクは翻訳済み画像を指すことがあります。
- `-img` を除外した場合: 画像リンクはソースの画像を指したままになることがあります。

すべての内部リンクが翻訳済みの出力を優先するようにするには、コンテンツ全体の翻訳を実行してください:

```bash
translate -l "ko" -md -nb -img
```

翻訳後にリンクレビューを実行してください:

```bash
co-op-review -l "ko"
```

## マークダウンのレンダリング問題

翻訳された Markdown が正しくレンダリングされない場合:

- フロントマターが `---` で開始および終了していることを確認してください。
- ソースと翻訳ファイル間でコードフェンスの数が一致していることを確認してください。
- 一般的な構造の問題を検出するために `co-op-review` を実行してください。
- 出力が破損している場合は、該当ファイルを再翻訳してください。

```bash
co-op-review -l "ko" --format github
```

## GitHub Actions は実行されたがプルリクエストが作成されなかった

`peter-evans/create-pull-request` がブランチがベースより進んでいないと報告する場合、ワークフローはコミットするファイルを見つけられませんでした。

考えられる原因:

- 翻訳実行が変更を生成しなかった。
- `.gitignore` が `translations/`、`translated_images/`、または翻訳済みノートブックを除外している。
- `add-paths` が生成された出力ディレクトリと一致していない。
- 翻訳ステップが早期に終了した。

対処法:

1. `translations/` または `translated_images/` に生成されたファイルが存在することを確認してください。
2. `.gitignore` が生成された出力を無視していないことを確認してください。
3. 一致する `add-paths` を使用してください:

   ```yaml
   with:
     add-paths: |
       translations/
       translated_images/
   ```

4. 一時的に翻訳コマンドにデバッグフラグを追加してください:

   ```bash
   translate -l "ko" -md -d -s
   ```

5. ワークフローの権限に次が含まれていることを確認してください:

   ```yaml
   permissions:
     contents: write
     pull-requests: write
   ```

## 翻訳品質

機械翻訳は人によるレビューが必要な場合があります。実験的な品質スコアリングや低信頼度の修復ワークフローを行いたい場合にのみ `evaluate` を使用してください。

!!! warning "Experimental"
    `evaluate` はルールベースおよび LLM ベースのチェックを使用することがあり、そのスコアリングモデルやメタデータの挙動が変更される可能性があります。ワークフローが変更に対応できる準備ができていない限り、必須の CI ゲートからは外しておいてください。

決定論的な CI チェックには代わりに `co-op-review` を使用してください。
