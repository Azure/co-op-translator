# ការដោះស្រាយបញ្ហា

ប្រើទំព័រនេះនៅពេលដែលដំណើរការបកប្រែបញ្ចប់ដោយមិនរំពឹងទុក ទទួលបរាជ័យក្នុងពេលកំណត់រចនាសម្ព័ន្ធ ឬបង្កើត​ចេញលទ្ធផលដែលត្រូវការត្រួតពិនិត្យ។

## ចាប់ផ្តើមនៅទីនេះ

1. រត់ពាក្យបញ្ជាផ្តោតមួយជាមុន ដូចជា `translate -l "ko" -md`.
2. ដាក់ `-d` ដើម្បីបង្ហាញកំណត់ហេតុកុងសូល (debug logs)។
3. ដាក់ `-s` ដើម្បីរក្សាកំណត់ហេតុនឹងនៅក្រោម `<root-dir>/logs/`។
4. រត់ `co-op-review` បន្ទាប់ពីការបកប្រែ ដើម្បីពិនិត្យភាពទាន់សម័យ រចនាសម្ព័ន្ធ និងតំណក្នុងស្រុក។

```bash
translate -l "ko" -md -d -s
co-op-review -l "ko"
```

## កំហុសកំណត់រចនាសម្ព័ន្ធ

### គ្មានអ្នកផ្តល់ម៉ូដែលភាសា

កំហុស:

```text
No language model configuration found.
```

ដោះស្រាយ:

- កំណត់ Azure OpenAI ឬ OpenAI។
- ផ្ទៀងផ្ទាត់ថាអថេរទាំងនោះមាននៅក្នុងបរិវេណដែលពាក្យបញ្ជាដំណើរការ។
- សម្រាប់ការប្រើប្រាស់ក្នុងស្រុក ទុកពួកវា​នៅក្នុង `.env` នៅក្នុង root នៃគម្រោង។

មើល [Configuration](configuration.md)។

### ការបកប្រែរូបភាពដោយគ្មាន Azure AI Vision

កំហុស:

```text
Image translation requested but Azure AI Service is not configured.
```

ដោះស្រាយ:

- បន្ថែម `AZURE_AI_SERVICE_API_KEY`។
- បន្ថែម `AZURE_AI_SERVICE_ENDPOINT`។
- ឬរត់ពាក្យបញ្ជាផ្ទាល់តែអត្ថបទ ដូចជា `translate -l "ko" -md`។

### កូនសោ ឬ គោលដៅ (Endpoint) មិនត្រឹមត្រូវ

រោលរោងអាចរួមមាន `401`, កំហុសសិទ្ធិដែលត្រូវបានសរសេរចុះ, ឬកំហុសចូលដំណើរការចំណុច។

ដោះស្រាយ:

- ផ្ទៀងផ្ទាត់ថាកូនសោចייכ់ជាផ្នែកនៃធនធាន Azure ដូចគ្នានឹង endpoint។
- ផ្ទៀងផ្ទាត់ថាធនធានគាំទ្រមុខងារ Vision ពេលប្រើ `-img`។
- ផ្ទៀងផ្ទាត់ឈ្មោះ Azure OpenAI deployment និង API version ស្របទៅនឹង deployment របស់អ្នក។
- រត់ជាមួយកំណត់ហេតុដាក់ទិន្នន័យ: `translate -l "ko" -md -d -s`។

## គ្មានឯកសារណាមួយត្រូវបានបកប្រែ

មូលហេតុទូទៅ:

- ជម្រើស (flags) ដែលបានជ្រើសមិនផ្គុំនឹងឯកសាររបស់អ្នក។
- មានឯកសារបកប្រែមុនស្រាប់។
- ឯកសារមូលដ្ឋាននៅក្រោមថតដែលបានច្រាញ់។
- ពាក្យបញ្ជាកំពុងដំណើរការពី root គម្រោងខុស។

ការត្រួតពិនិត្យ:

```bash
translate -l "ko" -md --dry-run
translate -l "ko" -nb --dry-run
translate -l "ko" -img --dry-run
```

ប្រើ `--root-dir` នៅពេលពាក្យបញ្ជាត្រូវដំណើរការពីក្រៅ root របស់គម្រោង។

## អាកប្បកិរិយាតំណភ្ជាប់មិនរំពឹងទុក

ការសរសេរថ្មីតំណអាស្រ័យលើប្រភេទមាតិកាដែលបានជ្រើស:

- `-nb` included: តំណ notebook អាចបង្ហាញទៅកាន់ notebook ដែលបានបកប្រែ។
- `-nb` excluded: តំណ notebook អាចនៅតែបង្ហាញទៅកាន់ notebooks ដើម។
- `-img` included: តំណរូបភាពអាចបង្ហាញទៅកាន់រូបភាពដែលបានបកប្រែ។
- `-img` excluded: តំណរូបភាពអាចនៅតែបង្ហាញទៅកាន់រូបភាពដើម។

រត់ការបកប្រែមាតិកាពេញលេញពេលដែលតំណក្នុងទាំងអស់គួរតែអាន​លទ្ធផលបកប្រែ៖

```bash
translate -l "ko" -md -nb -img
```

រត់ការពិនិត្យតំណក្រោយការបកប្រែ៖

```bash
co-op-review -l "ko"
```

## បញ្ហាការបង្ហាញ Markdown

បើ Markdown ដែលបានបកប្រែបង្ហាញខុស:

- ពិនិត្យថា frontmatter ចាប់ផ្តើម និងបញ្ចប់ដោយ `---`។
- ពិនិត្យថាចំនួន code fence ត្រូវគ្នារវាងឯកសារដើម និងឯកសារបកប្រែ។
- រត់ `co-op-review` ដើម្បីចាប់បញ្ហារៀបចំទូទៅ។
- បកប្រែឡើងវិញឯកសារពិសេសនោះ ប្រសិនបើលទ្ធផលខូចខាត។

```bash
co-op-review -l "ko" --format github
```

## GitHub Action បានរត់ ប៉ុន្តែ មិនបានបង្កើត Pull Request

បើ `peter-evans/create-pull-request` រាយការណ៍ថាខ្សែសាខាមិននៅខាងមុខ base នោះ workflow មិនបានរកឃើញឯកសារណាមួយសម្រាប់ commit។

ហេតុផលដែលអាចកើតមាន:

- ការរត់បកប្រែមិនបានបង្កើតការផ្លាស់ប្តូរណា។
- `.gitignore` ច្រានចោល `translations/`, `translated_images/`, ឬ notebooks ដែលបានបកប្រែ។
- `add-paths` មិនផ្គូផ្គងនឹងថតលទ្ធផលដែលបង្កើត។
- ជំហានបកប្រែបានចាកចេញមុនពេល។

ដោះស្រាយ:

1. ផ្ទៀងផ្ទាត់ថាឯកសារដែលបានបង្កើតមាននៅក្នុង `translations/` ឬ `translated_images/`។
2. ផ្ទៀងផ្ទាត់ថា `.gitignore` មិនបដិសេធលទ្ធផលដែលបានបង្កើត។
3. ប្រើ `add-paths` ដែលត្រូវគ្នា៖

   ```yaml
   with:
     add-paths: |
       translations/
       translated_images/
   ```

4. បន្ថែម flag ដើម្បីដាក់កំណត់ហេតុបណ្តោះអាសន្នទៅពាក្យបញ្ជាបកប្រែ៖

   ```bash
   translate -l "ko" -md -d -s
   ```

5. ផ្ទៀងផ្ទាត់សិទ្ធិ workflow រួមមាន៖

   ```yaml
   permissions:
     contents: write
     pull-requests: write
   ```

## គុណភាពការបកប្រែ

ការបកប្រែដោយម៉ាស៊ីនអាចត្រូវការការត្រួតពិនិត្យដោយមនុស្ស។ ប្រើ `evaluate` ទុកតែពេលដែលអ្នកចង់បានការវាយតម្លៃគុណភាពស្រាស្រាយនិងដំណើរការជួសជុលសម្រាប់ករណីដែលទំនុកចិត្តទាប។

!!! warning "Experimental"
    `evaluate` អាចប្រើការត្រួតពិនិត្យដោយច្បាប់ និងដោយ LLM បែបផ្សំបាន និងម៉ូដែលពិន្ទុ និងអាកប្បកិរិយាមេតាដាតា​អាចផ្លាស់ប្ដូរ។ ទុកវាឲ្យនៅខាងក្រៅនៃច្រក CI ដែលជាការព្យួរទាមទារ លុះត្រាតើ workflow របស់អ្នកបានរៀបចំពីការផ្លាស់ប្ដូរទាំងនេះ។

សម្រាប់ការត្រួតពិនិត្យ CI ដែលពឹងផ្អែកលើលទ្ធផលថេរ ប្រើ `co-op-review` ជំនួស។
