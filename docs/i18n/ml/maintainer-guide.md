# പരിപാലകൻ മാർഗ്ഗനിർദേശം

ഈ പേജ് API, CLI, மற்றும் ഡോക്യുമെന്റേഷൻ സൈറ്റ് എങ്ങനെ ചേർന്ന് പ്രവർത്തിക്കുന്നു എന്നതിന്റെ സംഗ്രഹമാണ്.

## സാർവജനിക API പരിധി

സ്റ്റേബിൾ Python API ഞങ്ങളിൽ നിന്നാണ് രഫ്തിരിക്കുന്നത്:

```python
co_op_translator.api
```

സാർവജനിക API ഉള്ളടക്ക പരിഭാഷ സഹായി, പാത പുനഃരാഖ്യാനം സഹായി, പ്രോജക്റ്റ് തന്ത്രവത്കരണം, റിവ്യൂ എന്നിവയിൽ ക്രമീകരിച്ചിരിക്കുന്നു:

```python
from co_op_translator.api import (
    ImageTranslationOptions,
    MarkdownTranslationOptions,
    NotebookTranslationOptions,
    run_review,
    run_translation,
    rewrite_markdown_paths,
    rewrite_notebook_paths,
    translate_image_content,
    translate_markdown_content,
    translate_notebook_content,
    translate_project,
)
```

പുതിയ സാർവജനിക APIകൾ ചേർക്കുമ്പോൾ അപ്ഡേറ്റ് ചെയ്യുക:

- `src/co_op_translator/api/__init__.py`
- `docs/api.md`
- ബന്ധപ്പെട്ട API ടെസ്റ്റുകൾ `tests/co_op_translator/` ഉള്ളിൽ, ഉദാഹരണത്തിന് `test_api.py` അല്ലെങ്കിൽ `test_review_api.py`

പ്രോജക്റ്റ് നേരിട്ട് പിന്തുണയ്ക്കാൻ ഉദ്ദേശിക്കുന്നില്ലെങ്കിൽ താഴ്ന്ന-തരത്തിലുള്ള `core` മോഡ്യൂളുകൾ സ്ഥിരം API ആയി രേഖപ്പെടുത്തുന്നതിൽ നിന്ന് ഒഴിവാവുക.

## CLI പ്രവേശന പോയിന്റുകൾ

പാക്കേജ് ഈ Poetry സ്ക്രിപ്റ്റുകൾ നിർവചിക്കുന്നു:

```toml
[tool.poetry.scripts]
translate = "co_op_translator.__main__:main"
evaluate = "co_op_translator.__main__:main"
migrate-links = "co_op_translator.__main__:main"
co-op-review = "co_op_translator.__main__:main"
co-op-translator-mcp = "co_op_translator.mcp.server:main"
```

`src/co_op_translator/__main__.py` സ്ക്രിപ്റ്റ് നാമം അനുസരിച്ച് ഡിസ്പാച്ച് ചെയ്യുന്നു:

- `translate` `co_op_translator.cli.translate.translate_command` എന്നതിനെ വിളിക്കുന്നു
- `evaluate` `co_op_translator.cli.evaluate.evaluate_command` എന്നതിനെ വിളിക്കുന്നു
- `migrate-links` `co_op_translator.cli.migrate_links.migrate_links_command` എന്നതിനെ വിളിക്കുന്നു
- `co-op-review` `co_op_translator.cli.review.review_command` എന്നതിനെ വിളിക്കുന്നു

`co-op-translator-mcp` `__main__.py` ഒഴിവാക്കി നേരിട്ടും `co_op_translator.mcp.server:main` നെ വിളിക്കുന്നു.

CLI ഓപ്ഷനുകൾ ചേർക്കുമ്പോൾ അല്ലെങ്കിൽ മാറ്റുമ്പോൾ അപ്ഡേറ്റ് ചെയ്യുക:

- ബന്ധപ്പെട്ട `src/co_op_translator/cli/*.py` കമാൻഡ്
- `docs/cli.md`
- പെരുമാറ്റം മാറുന്നുണ്ടെങ്കിൽ CLI-സംബന്ധമായ ടെസ്റ്റുകൾ

## MCP സെർവർ

MCP സെർവർ ഈ ഫയലിൽ നടപ്പിലാക്കിയിരിക്കുന്നു:

```python
co_op_translator.mcp.server
```

സെർവർ ലക്ഷ്യമിട്ടാണ് പബ്ലിക് Python API-നെ റാപ്പ് ചെയ്യുന്നത്, താഴ്ന്ന-തലത്തിലുള്ള `core` മോഡ്യൂളുകൾ നേരിട്ട് വിളിക്കാതെ. MCP ക്ലയന്റുകൾ, Python കോൾ ചെയ്യുന്നതും CLI-യും ഒരേ പെരുമാറ്റം പങ്കിട്ടുകൊണ്ടിരിക്കുന്നത് ഉറപ്പാക്കാൻ ഈ അതിർksiyon അങ്ങേയറ്റം പരിരക്ഷിച്ചിരിക്കുക.

MCP ടൂളുകൾ ചേർക്കുമ്പോൾ അല്ലെങ്കിൽ മാറ്റുമ്പോൾ അപ്ഡേറ്റ് ചെയ്യുക:

- `src/co_op_translator/mcp/server.py`
- `tests/co_op_translator/test_mcp_server.py`
- `docs/mcp.md`
- പൊതു API ഉപരിതലത്തിൽമാറ്റമുണ്ടെങ്കിൽ `docs/api.md`

റിപ്പോസിറ്ററി പരിഭാഷാ ഉപകരണങ്ങൾ MCP വഴியாக മോഡൽ-കോളബിളാണ്‌ കൂടതൽ ഫയലുകൾ എഴുതാൻ കഴിവുള്ളത്. കാരണം, കൂടാതെ ഡീഫോൾട്ട് ആയി `dry_run=True` ആയി വെക്കുകയും non-dry-run പ്രോജക്റ്റ് പരിഭാഷയ്ക്ക് മുൻപ് `confirm_write=True` ആവശ്യമാക്കുകയും ചെയ്യുക.

## പരിഭാഷാ പ്രവാഹം

ഉയർന്ന-തലത്തിലുള്ള പ്രോജക്റ്റ് പരിഭാഷാ പ്രവാഹം:

1. CLI ഓപ്ഷനുകൾ അല്ലെങ്കിൽ API പരാമീറ്ററുകൾ പാഴ്‌സ് ചെയ്യുക.
2. `LLMConfig` ഉപയോഗിച്ച് LLM കോൺഫിഗറേഷൻ സാധുത പരിശോദിക്കുക.
3. ഇമേജ് പരിഭാഷ തിരഞ്ഞെടുക്കപ്പെട്ടാൽ Azure AI Vision സാധുത പരിശോധിക്കുക.
4. ഭാഷാ കോഡുകൾ സാധാരണവത്കരിക്കുക.
5. പഴയഭാഷാ ഫോൾഡർ ആല്യാസുകൾ കണ്ടെത്തുക.
6. പരിഭാഷ വോല്യൂം അനുമാനിക്കുക.
7. ബാധകമെങ്കിൽ README ഭാഷ/കോർസ് വിഭാഗങ്ങൾ അപ്ഡേറ്റ് ചെയ്യുക.
8. പ്രോജക്റ്റ് പരിഭാഷ `ProjectTranslator`-നു പുർണ്ണക്കാര്യമാക്കുക.
9. `ProjectTranslator` ഫയൽ പ്രോസസിംഗിന് `TranslationManager`-നെ നിയോഗിക്കുന്നു.

`TranslationManager` ശ്രദ്ധ കേന്ദ്രീകരിച്ച ഫയൽ-ടൈപ്പ് മിക്സിനുകളിൽ നിന്നും യോജിപ്പിച്ചിരിക്കുന്നു:

- `ProjectMarkdownTranslationMixin` Markdown ഫയൽ വായനകൾ, ഉള്ളടക്ക പരിഭാഷ, പാത പുനഃരാഖ്യാനം, മെറ്റാഡേറ്റാ, ഡിസ്‌ക്ലെയിമറുകൾ, എഴുത്തുകൾ എന്നിവ കൈകാര്യം ചെയ്യുന്നു.
- `ProjectNotebookTranslationMixin` നോട്ട്‌ബുക്ക് ഫയൽ വായനകൾ, Markdown-സെല്‍ പരിഭാഷ, പാത പുനഃരാഖ്യാനം, മെറ്റാഡേറ്റാ, ഡിസ്‌ക്ലെയിമറുകൾ, എഴുത്തുകൾ എന്നിവ കൈകാര്യം ചെയ്യുന്നു.
- `ProjectImageTranslationMixin` ഇമേജ് കണ്ടെത്തൽ, ടെക്‌സ്‌റ്റ് എക്സ്ട്രാക്ഷൻ/പരിഭാഷ, റെൻഡർ ചെയ്ത ഇമേജ് എഴുത്തുകൾ, മെറ്റാഡേറ്റ എന്നിവ കൈകാര്യം ചെയ്യുന്നു.

താഴ്ന്ന-തലത്തിലുള്ള ഉള്ളടക്ക APIകൾ പ്രോജക്റ്റ് വർക്ക്‌ഫ്ലോ ഒഴിവാക്കുന്നു:

1. `translate_markdown_content` and `translate_notebook_content` മെമ്മറിയിൽ ഉള്ള ഉള്ളടക്കമാത്രം പരിഭാഷ ചെയ്യുന്നു.
2. `translate_image_content` ഒരു ഒറ്റ ഇമേജിലെ ടെക്‌സ്‌റ്റ് പരിഭാഷപ്പെടുത്തി ഒരു റെൻഡർചെയ്ത ഇമേജ് ഒബ്ജെക്റ്റ് റിട്ടേൺ ചെയ്യുന്നു.
3. `rewrite_markdown_paths` and `rewrite_notebook_paths` സ്പെസിഫിക് പോസ്റ്റ്-പ്രോസസ്സിംഗ് സഹായി. ഇവ പരിഭാഷയോ പ്രോജക്റ്റ് എഴുത്തോ ഒന്നും ചെയ്യാറില്ല.

## റിവ്യൂ പ്രവാഹം

നിശ്ചിതമായ റിവ്യു പ്രവാഹം:

1. CLI ഓപ്ഷനുകൾ അല്ലെങ്കിൽ API പരാമീറ്ററുകൾ പാഴ്‌സ് ചെയ്യുക.
2. അഭ്യർഥിച്ച ഭാഷാ കോഡുകൾ സാധാരണവത്കരിക്കുക.
3. `root_dir`, `root_dirs`, അല്ലെങ്കിൽ `groups` നിന്ന് ഒരു അല്ലെങ്കിൽ കൂടുതൽ റിവ്യൂ ലക്ഷ്യങ്ങൾ നിർമിക്കുക.
4. ഓപ്ഷണൽ ആയി `--changed-from` ഉപയോഗിച്ച് സോഴ്‌സ് ഫയലുകൾ പരിമിതപ്പെടുത്തുക.
5. ഘടന, പരിഭാഷ تازായത, Markdown ഏകീകരണം, പ്രാദേശിക ലിങ്ക്/ഇമേജ് പാതകൾ എന്നിവയ്ക്ക് നിശ്ചിതമായ പരിശോധനകൾ നടക്കുക.
6. ടെക്സ്റ്റ് ഔട്ട്പുട്ടോ GitHub-ശൈലിയിലുള്ള Markdown-ഉം പ്രിന്റ് ചെയ്യുക.
7. റിവ്യൂ പിശകുകൾ കണ്ടെത്തിയാൽ ഫെയിലർ കോഡോടെ പുറത്ത് നിൽക്കുക.

റിവ്യൂ പ്രവാഹത്തിനായി API കീകൾ ആവശ്യമാണ് എന്നില്ല; ഇത് പുൾ റിക്വസ്റ്റ് CI-യ്ക്ക് അനുയോജ്യമായിരിക്കണം. പുൾ റിക്വസ്റ്റ് വർക്ക്‌ഫ്ലോ ഓരോ റൺ-ലും ഒരു ചെക്ക് സംഗ്രഹം എഴുതുകയും `co-op-review` പരാജയപ്പെടുമ്പോഴേക്കും മാത്രമേ PR കമന്റ് പോസ്റ്റ് ചെയ്‌യുകയുള്ളൂ.

## ഡോക്യുമെന്റേഷൻ സൈറ്റ്

ഡോക്സ് സൈറ്റ് ഇതിലൂടെ കോൺഫിഗർ ചെയ്തിരിക്കുന്നു:

```text
mkdocs.yml
requirements-docs.txt
docs/
```

`docs/` ഡയറക്ടറി കാനോൺیکل‍ ഡോക്യുമെന്റേഷൻ ഉറവിടമാണ്. പ്രോജക്റ്റ് എന്നെറ്റ് മറ്റൊരു പ്രസിദ്ധീകരിച്ച ഡോക്യുമെന്റേഷൻ ഉപരിതലവുമായി ലക്ഷ്യം നൽകിയല്ലെങ്കിൽ ഈ ഡയറക്ടറിക്ക് പുറത്ത് പുതിയ end-user ഗൈഡുകൾ ചേർക്കരുത്.

ലൊക്കലായി ബിൽഡ് ചെയ്യുക:

```bash
python -m pip install -r requirements-docs.txt
python -m mkdocs build --strict
```

ലൊക്കലായി പ്രിവ്യൂ ചെയ്യുക:

```bash
python -m mkdocs serve
```

ജനറേറ്റു ചെയ്ത സൈറ്റ് `site/`-ലേക്ക് എഴുതപ്പെടുന്നു, അത് git വഴി അവഗണിക്കപ്പെട്ടിരിക്കുന്നു.

## GitHub Pages വർക്ക്‌ഫ്ലോ

.github/workflows/docs.yml പുൾ റിക്വസ്റ്റുകളിലും സൈറ്റ് ബിൽഡ് ചെയ്യുകയും `main`-ലേക്കുള്ള പുഷുകളിൽ അത് ഡിപ്ലോയ് ചെയ്യുകയും ചെയ്യുന്നു.

വർക്ക്ഫ്ലോ ഇൻസ്റ്റാൾ ചെയ്യുന്നത്:

```bash
pip install -r requirements-docs.txt
```

ഡോക്സ് വർക്‌ഫ്ലോ ഡോക്യുമെന്റേഷൻ ടൂൾചെയിൻ മാത്രമേ ഇൻസ്റ്റാൾ ചെയ്യൂ. `mkdocs.yml` `mkdocstrings`-നെ `src/` കണ്ടു നോട്ടം വെക്കുന്നു, അതിലൂടെ പബ്ലിക് API പേജുകൾ മൂല സ്രോതസ്സ് ട്രീ നിന്നുനിന്നാണോ റന്റർ ചെയ്യുന്നതെന്ന് നിങ്ങളുടെ റൺടൈം ഡിപൻഡൻസി സെറ്റ് ഇൻസ്റ്റാൾ ചെയ്യാതെ കാണിക്കാൻ കഴിയും. ഭവനത്തിൽ API ഡോക്സ് ബിൽഡിന്റെ സമയത്ത് ഓപ്ഷണൽ റൺടൈം പ്രൊവൈഡർമാരെ ഇറക്കുമതി ചെയ്യേണ്ടിവരുന്നതെങ്കിൽ `.github/workflows/docs.yml`-ഉം ഈ ഗൈഡും ഇക്കൊടൊപ്പം അപ്ഡേറ്റ് ചെയ്യുക.

ഡോക്യുമെന്റ് മാറ്റങ്ങൾ മേഴ്ജ് ചെയ്യുന്നതിന് മുൻപ്, ഓടിക്കുക:

```bash
python -m mkdocs build --strict
git diff --check
```

ഭദ്രമായ ബിൽഡുകൾ ഉപയോഗിക്കുക, jotta തകർന്ന ലിങ്കുകൾ, അസാധുവായ നാവിഗേഷൻ എൻട്രികൾ, API റെൻഡറിംഗ് പ്രശ്നങ്ങൾ തുടങ്ങിയവ ആദ്യം അറിയിച്ച് പരാജയപ്പെടും.
