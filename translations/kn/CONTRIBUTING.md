# Co-op Translator ಗೆ ಕೊಡುಗೆ ನೀಡುವುದು

ಈ ಯೋಜನೆ ಕೊಡುಗೆಗಳು ಮತ್ತು ಸಲಹೆಗಳನ್ನು ಸ್ವಾಗತಿಸುತ್ತದೆ. ಹೆಚ್ಚಿನ ಕೊಡುಗೆಗಳಿಗೆ ನೀವು Contributor License Agreement (CLA) ಒಪ್ಪಿಕೊಳ್ಳಬೇಕಾಗುತ್ತದೆ, ಇದು ನಿಮ್ಮ ಕೊಡುಗೆಯನ್ನು ಬಳಸಲು ನಮಗೆ ಹಕ್ಕುಗಳನ್ನು ನೀಡಲು ನೀವು ಹಕ್ಕು ಹೊಂದಿದ್ದೀರಿ ಮತ್ತು ನಿಜವಾಗಿಯೂ ನೀಡುತ್ತೀರಿ ಎಂದು ಘೋಷಿಸುತ್ತದೆ. ವಿವರಗಳಿಗೆ, https://cla.opensource.microsoft.com ಗೆ ಭೇಟಿ ನೀಡಿ.

ನೀವು pull request ಸಲ್ಲಿಸಿದಾಗ, CLA ಬಾಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನೀವು CLA ಒದಗಿಸಬೇಕೇ ಎಂದು ನಿರ್ಧರಿಸುತ್ತದೆ ಮತ್ತು PR ಅನ್ನು ಸೂಕ್ತವಾಗಿ ಅಲಂಕರಿಸುತ್ತದೆ (ಉದಾ., ಸ್ಥಿತಿಯ ಪರಿಶೀಲನೆ, ಕಾಮೆಂಟ್). ಬಾಟ್ ಒದಗಿಸಿದ ಸೂಚನೆಗಳನ್ನು ಅನುಸರಿಸಿ. ನಮ್ಮ CLA ಬಳಸುವ ಎಲ್ಲಾ reposಗಳಲ್ಲಿ ನೀವು ಇದನ್ನು ಒಂದೇ ಬಾರಿ ಮಾಡಬೇಕಾಗುತ್ತದೆ.

## ಡೆವಲಪ್ಮೆಂಟ್ ಪರಿಸರವನ್ನು ಸೆಟಪ್ ಮಾಡುವುದು

ಈ ಯೋಜನೆಗೆ ಡೆವಲಪ್ಮೆಂಟ್ ಪರಿಸರವನ್ನು ಸೆಟಪ್ ಮಾಡಲು, dependencies ನಿರ್ವಹಿಸಲು Poetry ಬಳಸುವಂತೆ ನಾವು ಶಿಫಾರಸು ಮಾಡುತ್ತೇವೆ. ನಾವು `pyproject.toml` ಅನ್ನು ಯೋಜನೆಯ dependencies ನಿರ್ವಹಿಸಲು ಬಳಸುತ್ತೇವೆ, ಆದ್ದರಿಂದ dependencies ಅನ್ನು ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡಲು ನೀವು Poetry ಬಳಸಬೇಕು.

### ವರ್ಚುವಲ್ ಪರಿಸರವನ್ನು ರಚಿಸಿ

#### pip ಬಳಸಿ

```bash
python -m venv .venv
```

#### Poetry ಬಳಸಿ

```bash
poetry init
```

### ವರ್ಚುವಲ್ ಪರಿಸರವನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಿ

#### pip ಮತ್ತು Poetry ಎರಡರಿಗೂ

- ವಿಂಡೋಸ್:

    ```bash
    .venv\Scripts\activate.bat
    ```

- ಮ್ಯಾಕ್/ಲಿನಕ್ಸ್:

    ```bash
    source .venv/bin/activate
    ```

#### Poetry ಬಳಸಿ

```bash
poetry shell
```

### ಪ್ಯಾಕೇಜ್ ಮತ್ತು ಅಗತ್ಯ ಪ್ಯಾಕೇಜ್‌ಗಳನ್ನು ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡುವುದು

#### Poetry ಬಳಸಿ (pyproject.toml ನಿಂದ)

```bash
poetry install
```

### ಕೈಯಿಂದ ಪರೀಕ್ಷೆ

PR ಸಲ್ಲಿಸುವ ಮೊದಲು, ನಿಜವಾದ ಡಾಕ್ಯುಮೆಂಟೇಶನ್‌ನೊಂದಿಗೆ ಅನುವಾದ ಕಾರ್ಯಕ್ಷಮತೆಯನ್ನು ಪರೀಕ್ಷಿಸುವುದು ಮುಖ್ಯ:

1. ಮೂಲ ಡೈರೆಕ್ಟರಿಯಲ್ಲಿ ಒಂದು ಪರೀಕ್ಷಾ ಡೈರೆಕ್ಟರಿಯನ್ನು ರಚಿಸಿ:
    ```bash
    mkdir test_docs
    ```

2. ನೀವು ಅನುವಾದಿಸಲು ಬಯಸುವ ಕೆಲವು markdown ಡಾಕ್ಯುಮೆಂಟೇಶನ್ ಮತ್ತು ಚಿತ್ರಗಳನ್ನು ಪರೀಕ್ಷಾ ಡೈರೆಕ್ಟರಿಗೆ ನಕಲು ಮಾಡಿ. ಉದಾಹರಣೆಗೆ:
    ```bash
    cp /path/to/your/docs/*.md test_docs/
    cp /path/to/your/images/*.png test_docs/
    ```

3. ಪ್ಯಾಕೇಜ್ ಅನ್ನು ಸ್ಥಳೀಯವಾಗಿ ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡಿ:
    ```bash
    pip install -e .
    ```

4. ನಿಮ್ಮ ಪರೀಕ್ಷಾ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳಲ್ಲಿ Co-op Translator ಅನ್ನು ಚಲಾಯಿಸಿ:
    ```bash
    python -m co_op_translator --language-codes ko --root-dir test_docs
    ```

5. `test_docs/translations` ಮತ್ತು `test_docs/translated_images` ನಲ್ಲಿ ಅನುವಾದಿತ ಫೈಲ್‌ಗಳನ್ನು ಪರಿಶೀಲಿಸಿ:
   - ಅನುವಾದದ ಗುಣಮಟ್ಟ
   - ಮೆಟಾಡೇಟಾ ಕಾಮೆಂಟ್‌ಗಳು ಸರಿಯಾಗಿವೆ
   - ಮೂಲ markdown ರಚನೆ ಉಳಿಸಲಾಗಿದೆ
   - ಲಿಂಕ್‌ಗಳು ಮತ್ತು ಚಿತ್ರಗಳು ಸರಿಯಾಗಿ ಕೆಲಸ ಮಾಡುತ್ತಿವೆ

ಈ ಕೈಯಿಂದ ಪರೀಕ್ಷೆ ನಿಮ್ಮ ಬದಲಾವಣೆಗಳು ನಿಜವಾದ ಪರಿಸ್ಥಿತಿಗಳಲ್ಲಿ ಚೆನ್ನಾಗಿ ಕೆಲಸ ಮಾಡುತ್ತವೆ ಎಂಬುದನ್ನು ಖಚಿತಪಡಿಸುತ್ತದೆ.

### ಪರಿಸರ ವ್ಯತ್ಯಯಗಳು

1. ಒದಗಿಸಿದ `.env.template` ಫೈಲ್ ಅನ್ನು ನಕಲು ಮಾಡುವ ಮೂಲಕ ಮೂಲ ಡೈರೆಕ್ಟರಿಯಲ್ಲಿ `.env` ಫೈಲ್ ಅನ್ನು ರಚಿಸಿ.
1. ಮಾರ್ಗದರ್ಶನದಂತೆ ಪರಿಸರ ವ್ಯತ್ಯಯಗಳನ್ನು ತುಂಬಿ.

### ಕೋಡ್ ಶೈಲಿ

ನಾವು [Black](https://github.com/psf/black) ಅನ್ನು ನಮ್ಮ Python ಕೋಡ್ ಫಾರ್ಮ್ಯಾಟರ್ ಆಗಿ ಬಳಸುತ್ತೇವೆ, ಇದು ಯೋಜನೆಯಾದ್ಯಂತ ಸತತ ಕೋಡ್ ಶೈಲಿಯನ್ನು ನಿರ್ವಹಿಸಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ. Black ಒಂದು uncompromising ಕೋಡ್ ಫಾರ್ಮ್ಯಾಟರ್ ಆಗಿದ್ದು, Python ಕೋಡ್ ಅನ್ನು Black ಕೋಡ್ ಶೈಲಿಗೆ ಅನುಗುಣವಾಗಿ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಮರುರೂಪಿಸುತ್ತದೆ.

#### ಸಂರಚನೆ

Black ಸಂರಚನೆ ನಮ್ಮ `pyproject.toml` ನಲ್ಲಿ ನಿರ್ದಿಷ್ಟವಾಗಿದೆ:

```toml
[tool.black]
line-length = 88
target-version = ['py310']
include = '\.pyi?$'
```

#### Black ಅನ್ನು ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡುವುದು

ನೀವು Poetry (ಶಿಫಾರಸು) ಅಥವಾ pip ಬಳಸಿ Black ಅನ್ನು ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡಬಹುದು:

##### Poetry ಬಳಸಿ

Black ಅನ್ನು ನೀವು ಡೆವಲಪ್ಮೆಂಟ್ ಪರಿಸರವನ್ನು ಸೆಟಪ್ ಮಾಡಿದಾಗ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡಲಾಗುತ್ತದೆ:
```bash
poetry install
```

##### pip ಬಳಸಿ

ನೀವು pip ಬಳಸುತ್ತಿದ್ದರೆ, ನೀವು Black ಅನ್ನು ನೇರವಾಗಿ ಇನ್‌ಸ್ಟಾಲ್ ಮಾಡಬಹುದು:
```bash
pip install black
```

#### Black ಅನ್ನು ಬಳಸುವುದು

##### Poetry ಬಳಸಿ

1. ಯೋಜನೆಯಲ್ಲಿರುವ ಎಲ್ಲಾ Python ಫೈಲ್‌ಗಳನ್ನು ಫಾರ್ಮ್ಯಾಟ್ ಮಾಡಿ:
    ```bash
    poetry run black .
    ```

2. ನಿರ್ದಿಷ್ಟ ಫೈಲ್ ಅಥವಾ ಡೈರೆಕ್ಟರಿಯನ್ನು ಫಾರ್ಮ್ಯಾಟ್ ಮಾಡಿ:
    ```bash
    poetry run black path/to/file_or_directory
    ```

##### pip ಬಳಸಿ

1. ಯೋಜನೆಯಲ್ಲಿರುವ ಎಲ್ಲಾ Python ಫೈಲ್‌ಗಳನ್ನು ಫಾರ್ಮ್ಯಾಟ್ ಮಾಡಿ:
    ```bash
    black .
    ```

2. ನಿರ್ದಿಷ್ಟ ಫೈಲ್ ಅಥವಾ ಡೈರೆಕ್ಟರಿಯನ್ನು ಫಾರ್ಮ್ಯಾಟ್ ಮಾಡಿ:
    ```bash
    black path/to/file_or_directory
    ```

> [!TIP]
> Black ಅನ್ನು save ಮಾಡಿದಾಗ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಕೋಡ್ ಅನ್ನು ಫಾರ್ಮ್ಯಾಟ್ ಮಾಡಲು ನಿಮ್ಮ ಎಡಿಟರ್ ಅನ್ನು ಸೆಟಪ್ ಮಾಡುವಂತೆ ನಾವು ಶಿಫಾರಸು ಮಾಡುತ್ತೇವೆ. ಹೆಚ್ಚಿನ ಆಧುನಿಕ ಎಡಿಟರ್‌ಗಳು ಇದನ್ನು extensions ಅಥವಾ plugins ಮೂಲಕ ಬೆಂಬಲಿಸುತ್ತವೆ.

## Co-op Translator ಅನ್ನು ಚಲಾಯಿಸುವುದು

Poetry ಬಳಸಿ ನಿಮ್ಮ ಪರಿಸರದಲ್ಲಿ Co-op Translator ಅನ್ನು ಚಲಾಯಿಸಲು, ಈ ಹಂತಗಳನ್ನು ಅನುಸರಿಸಿ:

1. ನೀವು ಅನುವಾದ ಪರೀಕ್ಷೆಗಳನ್ನು ನಡೆಸಲು ಬಯಸುವ ಡೈರೆಕ್ಟರಿಗೆ ಹೋಗಿ ಅಥವಾ ತಾತ್ಕಾಲಿಕ ಫೋಲ್ಡರ್ ಅನ್ನು ರಚಿಸಿ.

2. ಈ ಕೆಳಗಿನ ಆಜ್ಞೆಯನ್ನು ಚಲಾಯಿಸಿ. `-l ko` ಅನ್ನು ನೀವು ಅನುವಾದಿಸಲು ಬಯಸುವ ಭಾಷಾ ಕೋಡ್‌ನೊಂದಿಗೆ ಬದಲಾಯಿಸಿ. `-d` ಫ್ಲಾಗ್ ಡೀಬಗ್ ಮೋಡ್ ಅನ್ನು ಸೂಚಿಸುತ್ತದೆ.

    ```bash
    poetry run co-op-translator translate -l ko -d
    ```

> [!NOTE]
> ಆಜ್ಞೆಯನ್ನು ಚಲಾಯಿಸುವ ಮೊದಲು ನಿಮ್ಮ Poetry ಪರಿಸರ ಸಕ್ರಿಯಗೊಳಿಸಲಾಗಿದೆ (poetry shell) ಎಂದು ಖಚಿತಪಡಿಸಿ.

## ಹೊಸ ಭಾಷೆಯನ್ನು ಕೊಡುಗೆ ನೀಡುವುದು

ಹೊಸ ಭಾಷೆಗಳಿಗೆ ಬೆಂಬಲವನ್ನು ಸೇರಿಸುವ ಕೊಡುಗೆಗಳನ್ನು ನಾವು ಸ್ವಾಗತಿಸುತ್ತೇವೆ. PR ತೆರೆಯುವ ಮೊದಲು, smoother review ಖಚಿತಪಡಿಸಲು ಕೆಳಗಿನ ಹಂತಗಳನ್ನು ಪೂರ್ಣಗೊಳಿಸಿ.

1. ಭಾಷೆಯನ್ನು ಫಾಂಟ್ ಮ್ಯಾಪಿಂಗ್‌ಗೆ ಸೇರಿಸಿ
   - `src/co_op_translator/fonts/font_language_mappings.yml` ಅನ್ನು ಸಂಪಾದಿಸಿ
   - ಈ ವಿವರಗಳೊಂದಿಗೆ ಒಂದು ಎಂಟ್ರಿಯನ್ನು ಸೇರಿಸಿ:
     - `code`: ISO-ಹೋಲುವ ಭಾಷಾ ಕೋಡ್ (ಉದಾ., `vi`)
     - `name`: ಮಾನವ ಸ್ನೇಹಿ ಪ್ರದರ್ಶನ ಹೆಸರು
     - `font`: `src/co_op_translator/fonts/` ನಲ್ಲಿ script ಅನ್ನು ಬೆಂಬಲಿಸುವ ಫಾಂಟ್
     - `rtl`: ಬಲದಿಂದ ಎಡಕ್ಕೆ ಇದ್ದರೆ `true`, ಇಲ್ಲವಾದರೆ `false`

2. ಅಗತ್ಯವಿರುವ ಫಾಂಟ್ ಫೈಲ್‌ಗಳನ್ನು ಸೇರಿಸಿ (ಅಗತ್ಯವಿದ್ದರೆ)
   - ಹೊಸ ಫಾಂಟ್ ಅಗತ್ಯವಿದ್ದರೆ, ಓಪನ್ ಸೋರ್ಸ್ ವಿತರಣೆಗೆ ಪರವಾನಗಿ ಹೊಂದಾಣಿಕೆಯನ್ನು ಪರಿಶೀಲಿಸಿ
   - ಫಾಂಟ್ ಫೈಲ್ ಅನ್ನು `src/co_op_translator/fonts/` ಗೆ ಸೇರಿಸಿ

3. ಸ್ಥಳೀಯ ದೃಢೀಕರಣ
   - ಸಣ್ಣ ಮಾದರಿಗಾಗಿ ಅನುವಾದಗಳನ್ನು ಚಲಾಯಿಸಿ (Markdown, ಚಿತ್ರಗಳು, ಮತ್ತು notebooks ಅಗತ್ಯವಿದ್ದಂತೆ)
   - ಔಟ್‌ಪುಟ್ ಸರಿಯಾಗಿ ರೆಂಡರ್ ಆಗುತ್ತದೆಯೇ ಎಂದು ಪರಿಶೀಲಿಸಿ, ಫಾಂಟ್‌ಗಳು ಮತ್ತು RTL ಲೇಔಟ್ ಸೇರಿ

4. ಡಾಕ್ಯುಮೆಂಟೇಶನ್ ಅನ್ನು ಅಪ್ಡೇಟ್ ಮಾಡಿ
   - ಭಾಷೆ `getting_started/supported-languages.md` ನಲ್ಲಿ ಕಾಣಿಸುತ್ತದೆ ಎಂದು ಖಚಿತಪಡಿಸಿ
   - `getting_started/README_languages_template.md` ಗೆ ಯಾವುದೇ ಬದಲಾವಣೆಗಳ ಅಗತ್ಯವಿಲ್ಲ; ಇದು ಬೆಂಬಲಿತ ಪಟ್ಟಿದಿಂದ ತಯಾರಿಸಲಾಗುತ್ತದೆ

5. PR ತೆರೆಯಿರಿ
   - ಸೇರಿಸಿದ ಭಾಷೆಯನ್ನು ಮತ್ತು ಯಾವುದೇ ಫಾಂಟ್/ಪರವಾನಗಿ ಪರಿಗಣನೆಗಳನ್ನು ವಿವರಿಸಿ
   - ಸಾಧ್ಯವಾದರೆ ರೆಂಡರ್ ಮಾಡಿದ ಔಟ್‌ಪುಟ್‌ಗಳ ಸ್ಕ್ರೀನ್‌ಶಾಟ್‌ಗಳನ್ನು ಲಗತ್ತಿಸಿ

Example YAML entry:

```yaml
new_lang(code):
  name: "New Language"
  font: "NotoSans-Medium.ttf"
  rtl: false
```

### ಹೊಸ ಭಾಷೆಯನ್ನು ಪರೀಕ್ಷಿಸಿ

ನೀವು ಈ ಕೆಳಗಿನ ಆಜ್ಞೆಯನ್ನು ಚಲಾಯಿಸುವ ಮೂಲಕ ಹೊಸ ಭಾಷೆಯನ್ನು ಪರೀಕ್ಷಿಸಬಹುದು:

```bash
# ವರ್ಚುವಲ್ ಪರಿಸರವನ್ನು ರಚಿಸಿ ಮತ್ತು ಸಕ್ರಿಯಗೊಳಿಸಿ
python -m venv .venv
# ವಿಂಡೋಸ್
.venv\Scripts\activate
# ಮ್ಯಾಕ್‌ಒಎಸ್/ಲಿನಕ್ಸ್
source .venv/bin/activate
# ಅಭಿವೃದ್ಧಿ ಪ್ಯಾಕೇಜ್ ಅನ್ನು ಸ್ಥಾಪಿಸಿ
pip install -e .
# ಅನುವಾದವನ್ನು ಚಲಾಯಿಸಿ
translate -l "new_lang"
```

## ನಿರ್ವಹಕರು

### Commit ಸಂದೇಶ ಮತ್ತು Merge ತಂತ್ರ

ನಮ್ಮ ಯೋಜನೆಯ commit ಇತಿಹಾಸದಲ್ಲಿ ಸತತತೆ ಮತ್ತು ಸ್ಪಷ್ಟತೆಯನ್ನು ಖಚಿತಪಡಿಸಲು, **Squash and Merge** ತಂತ್ರವನ್ನು ಬಳಸುವಾಗ **ಅಂತಿಮ commit ಸಂದೇಶ**ಕ್ಕಾಗಿ ನಾವು ನಿರ್ದಿಷ್ಟ commit ಸಂದೇಶ ಸ್ವರೂಪವನ್ನು ಅನುಸರಿಸುತ್ತೇವೆ.

Pull request (PR) ಅನ್ನು ಮರ್ಜ್ ಮಾಡಿದಾಗ, ವೈಯಕ್ತಿಕ commit‌ಗಳನ್ನು ಒಂದೇ commit‌ಗೆ squash ಮಾಡಲಾಗುತ್ತದೆ. ಅಂತಿಮ commit ಸಂದೇಶವು ಸ್ವರೂಪವನ್ನು ಅನುಸರಿಸಬೇಕು ಎಂದು ಖಚಿತಪಡಿಸಲು ಕೆಳಗಿನ ಸ್ವರೂಪವನ್ನು ಅನುಸರಿಸುತ್ತೇವೆ.

#### Commit ಸಂದೇಶ ಸ್ವರೂಪ (Squash and Merge ಗೆ)

ನಾವು commit ಸಂದೇಶಗಳಿಗೆ ಈ ಸ್ವರೂಪವನ್ನು ಬಳಸುತ್ತೇವೆ:

```bash
<type>: <description> (#<ಪಿಆರ್ ಸಂಖ್ಯೆ>)
```

- **type**: commit ವರ್ಗವನ್ನು ನಿರ್ದಿಷ್ಟಪಡಿಸುತ್ತದೆ. ನಾವು ಈ ಕೆಳಗಿನ ಪ್ರಕಾರಗಳನ್ನು ಬಳಸುತ್ತೇವೆ:
  - `Docs`: ಡಾಕ್ಯುಮೆಂಟೇಶನ್ ಅಪ್ಡೇಟ್‌ಗಳಿಗೆ.
  - `Build`: build ವ್ಯವಸ್ಥೆ ಅಥವಾ dependencies ಸಂಬಂಧಿತ ಬದಲಾವಣೆಗಳಿಗೆ, configuration ಫೈಲ್‌ಗಳು, CI workflows, ಅಥವಾ Dockerfile ಅಪ್ಡೇಟ್‌ಗಳನ್ನು ಒಳಗೊಂಡಂತೆ.
  - `Core`: `src/co_op_translator/core` ಡೈರೆಕ್ಟರಿಯಲ್ಲಿರುವ ಫೈಲ್‌ಗಳನ್ನು ಒಳಗೊಂಡಂತೆ, ಯೋಜನೆಯ ಮೂಲ ಕಾರ್ಯಕ್ಷಮತೆ ಅಥವಾ ವೈಶಿಷ್ಟ್ಯಗಳಲ್ಲಿ ಬದಲಾವಣೆಗಳಿಗೆ.

- **description**: ಬದಲಾವಣೆಯ ಸಂಕ್ಷಿಪ್ತ ಸಾರಾಂಶ.
- **PR number**: commit ಗೆ ಸಂಬಂಧಿಸಿದ pull request ಸಂಖ್ಯೆ.

**ಉದಾಹರಣೆಗಳು**:

- `Docs: Update installation instructions for clarity (#50)`
- `Core: Improve handling of image translation (#60)`

> [!NOTE]
> ಪ್ರಸ್ತುತ, **`Docs`**, **`Core`**, ಮತ್ತು **`Build`** ಪ್ರೀಫಿಕ್ಸ್‌ಗಳನ್ನು PR ಶೀರ್ಷಿಕೆಗಳಿಗೆ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಸೇರಿಸಲಾಗುತ್ತದೆ, ಬದಲಾಯಿಸಿದ ಮೂಲ ಕೋಡ್‌ಗೆ ಲೇಬಲ್‌ಗಳನ್ನು ಅನ್ವಯಿಸಿದ ಆಧಾರದ ಮೇಲೆ. ಸರಿಯಾದ ಲೇಬಲ್ ಅನ್ವಯಿಸಲಾಗಿದೆ ಎಂದು ನೀವು ಖಚಿತಪಡಿಸಿಕೊಳ್ಳಬೇಕಾಗಿದೆ, ಮತ್ತು ಪ್ರೀಫಿಕ್ಸ್ ಸರಿಯಾಗಿ ತಯಾರಿಸಲಾಗಿದೆ ಎಂದು ಪರಿಶೀಲಿಸಬೇಕು.

#### Merge ತಂತ್ರ

ನಾವು **Squash and Merge** ಅನ್ನು pull requests ಗೆ ಡೀಫಾಲ್ಟ್ ತಂತ್ರವಾಗಿ ಬಳಸುತ್ತೇವೆ. ಈ ತಂತ್ರವು commit ಸಂದೇಶಗಳು ನಮ್ಮ ಸ್ವರೂಪವನ್ನು ಅನುಸರಿಸುತ್ತವೆ ಎಂದು ಖಚಿತಪಡಿಸುತ್ತದೆ, ವೈಯಕ್ತಿಕ commit‌ಗಳು ಅನುಸರಿಸದಿದ್ದರೂ.

**ಕಾರಣಗಳು**:

- ಸ್ವಚ್ಛ, ರೇಖೀಯ ಯೋಜನೆ ಇತಿಹಾಸ.
- commit ಸಂದೇಶಗಳಲ್ಲಿ ಸತತತೆ.
- ಸಣ್ಣ commit‌ಗಳಿಂದ (ಉದಾ., "fix typo") ಶಬ್ದವನ್ನು ಕಡಿಮೆ ಮಾಡಲಾಗಿದೆ.

ಮರ್ಜ್ ಮಾಡುವಾಗ, ಅಂತಿಮ commit ಸಂದೇಶವು ಮೇಲಿನ commit ಸಂದೇಶ ಸ್ವರೂಪವನ್ನು ಅನುಸರಿಸುತ್ತದೆ ಎಂದು ಖಚಿತಪಡಿಸಿ.

**Squash and Merge ಉದಾಹರಣೆ**
ಒಂದು PR ಈ commit‌ಗಳನ್ನು ಹೊಂದಿದ್ದರೆ:

- `fix typo`
- `update README`
- `adjust formatting`

ಅವುಗಳನ್ನು squash ಮಾಡಬೇಕು:
`Docs: Improve documentation clarity and formatting (#65)`
