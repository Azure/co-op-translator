## డెవలప్‌మెంట్ ఎన్విరాన్‌మెంట్ సెటప్

ఈ ప్రాజెక్ట్ కోసం డెవలప్‌మెంట్ ఎన్విరాన్‌మెంట్ సెటప్ చేయడానికి, డిపెండెన్సీలను నిర్వహించడానికి Poetry ఉపయోగించమని మేము సిఫార్సు చేస్తాము. మేము `pyproject.toml` ను ప్రాజెక్ట్ డిపెండెన్సీలను నిర్వహించడానికి ఉపయోగిస్తాము, అందువల్ల డిపెండెన్సీలను ఇన్‌స్టాల్ చేయడానికి మీరు Poetry ఉపయోగించాలి.

### వర్చువల్ ఎన్విరాన్‌మెంట్ క్రియేట్ చేయండి

#### pip ఉపయోగించి

```bash
python -m venv .venv
```

#### Poetry ఉపయోగించి

```bash
poetry init
```

### వర్చువల్ ఎన్విరాన్‌మెంట్ యాక్టివేట్ చేయండి

#### pip మరియు Poetry కోసం

- Windows:

    ```bash
    .venv\Scripts\activate.bat
    ```

- Mac/Linux:

    ```bash
    source .venv/bin/activate
    ```

#### Poetry ఉపయోగించి

```bash
poetry shell
```

### ప్యాకేజీ మరియు అవసరమైన ప్యాకేజీలను ఇన్‌స్టాల్ చేయడం

#### Poetry ఉపయోగించి (pyproject.toml నుండి)

```bash
poetry install
```

### మాన్యువల్ టెస్టింగ్

PR సమర్పించే ముందు, నిజమైన డాక్యుమెంటేషన్‌తో అనువాద ఫంక్షనాలిటీని పరీక్షించడం ముఖ్యం:

1. రూట్ డైరెక్టరీలో ఒక టెస్ట్ డైరెక్టరీని క్రియేట్ చేయండి:
    ```bash
    mkdir test_docs
    ```

2. మీరు అనువదించాలనుకుంటున్న కొన్ని markdown డాక్యుమెంటేషన్ మరియు చిత్రాలను టెస్ట్ డైరెక్టరీలో కాపీ చేయండి. ఉదాహరణకు:
    ```bash
    cp /path/to/your/docs/*.md test_docs/
    cp /path/to/your/images/*.png test_docs/
    ```

3. ప్యాకేజీని స్థానికంగా ఇన్‌స్టాల్ చేయండి:
    ```bash
    pip install -e .
    ```

4. మీ టెస్ట్ డాక్యుమెంట్లపై Co-op Translator ను రన్ చేయండి:
    ```bash
    python -m co_op_translator --language-codes ko --root-dir test_docs
    ```

5. `test_docs/translations` మరియు `test_docs/translated_images` లో అనువదించిన ఫైళ్లను తనిఖీ చేయండి:
   - అనువాద నాణ్యత
   - మెటాడేటా కామెంట్లు సరైనవా
   - అసలు markdown నిర్మాణం కాపాడబడిందా
   - లింకులు మరియు చిత్రాలు సరిగ్గా పనిచేస్తున్నాయా

ఈ మాన్యువల్ టెస్టింగ్ మీ మార్పులు నిజమైన ప్రపంచ పరిస్థితుల్లో బాగా పనిచేస్తున్నాయా అని నిర్ధారించడంలో సహాయపడుతుంది.

### ఎన్విరాన్‌మెంట్ వేరియబుల్స్

### కోడ్ స్టైల్

మేము ప్రాజెక్ట్ అంతటా స్థిరమైన కోడ్ స్టైల్‌ను నిర్వహించడానికి Python కోడ్ ఫార్మాటర్ [Black](https://github.com/psf/black) ను ఉపయోగిస్తాము. Black అనేది ఒక uncompromising కోడ్ ఫార్మాటర్, ఇది Black కోడ్ స్టైల్‌కు అనుగుణంగా Python కోడ్‌ను స్వయంచాలకంగా రీఫార్మాట్ చేస్తుంది.

#### కాన్ఫిగరేషన్

Black కాన్ఫిగరేషన్ మా `pyproject.toml` లో స్పెసిఫై చేయబడింది:

```toml
[tool.black]
line-length = 88
target-version = ['py310']
include = '\.pyi?$'
```

#### Black ను ఇన్‌స్టాల్ చేయడం

మీరు Poetry (సిఫార్సు) లేదా pip ఉపయోగించి Black ను ఇన్‌స్టాల్ చేయవచ్చు:

##### Poetry ఉపయోగించి

Black డెవలప్‌మెంట్ ఎన్విరాన్‌మెంట్ సెటప్ చేసినప్పుడు స్వయంచాలకంగా ఇన్‌స్టాల్ చేయబడుతుంది:
```bash
poetry install
```

##### pip ఉపయోగించి

మీరు pip ఉపయోగిస్తే, మీరు Black ను నేరుగా ఇన్‌స్టాల్ చేయవచ్చు:
```bash
pip install black
```

#### Black ఉపయోగించడం

##### Poetry తో

1. ప్రాజెక్ట్‌లోని అన్ని Python ఫైళ్లను ఫార్మాట్ చేయండి:
    ```bash
    poetry run black .
    ```

2. ఒక నిర్దిష్ట ఫైల్ లేదా డైరెక్టరీని ఫార్మాట్ చేయండి:
    ```bash
    poetry run black path/to/file_or_directory
    ```

##### pip తో

1. ప్రాజెక్ట్‌లోని అన్ని Python ఫైళ్లను ఫార్మాట్ చేయండి:
    ```bash
    black .
    ```

2. ఒక నిర్దిష్ట ఫైల్ లేదా డైరెక్టరీని ఫార్మాట్ చేయండి:
    ```bash
    black path/to/file_or_directory
    ```

> [!TIP]
> Black తో కోడ్‌ను సేవ్ చేసినప్పుడు ఆటోమేటిక్‌గా ఫార్మాట్ చేయడానికి మీ ఎడిటర్‌ను సెటప్ చేయమని మేము సిఫార్సు చేస్తాము. ఆధునిక ఎడిటర్లు ఎక్కువగా ఈ ఫీచర్‌ను extensions లేదా plugins ద్వారా సపోర్ట్ చేస్తాయి.

## Co-op Translator ను రన్ చేయడం

మీ ఎన్విరాన్‌మెంట్‌లో Poetry ఉపయోగించి Co-op Translator ను రన్ చేయడానికి, ఈ స్టెప్స్‌ను పాటించండి:

1. మీరు అనువాద టెస్టులు చేయాలనుకుంటున్న డైరెక్టరీకి వెళ్లండి లేదా టెస్టింగ్ కోసం తాత్కాలిక ఫోల్డర్ క్రియేట్ చేయండి.

2. ఈ కింది కమాండ్‌ను అమలు చేయండి. మీరు అనువదించాలనుకుంటున్న భాష కోడ్‌ను `-l ko` తో మార్చండి. `-d` ఫ్లాగ్ డీబగ్ మోడ్‌ను సూచిస్తుంది.

    ```bash
    poetry run co-op-translator translate -l ko -d
    ```

> [!NOTE]
> కమాండ్‌ను రన్ చేయడానికి ముందు మీ Poetry ఎన్విరాన్‌మెంట్ యాక్టివేట్ చేయబడిందని (poetry shell) నిర్ధారించండి.

## కొత్త భాషను కాంట్రిబ్యూట్ చేయండి

కొత్త భాషలకు మద్దతు జోడించే సహకారాలను మేము స్వాగతిస్తున్నాము. PR ఓపెన్ చేయడానికి ముందు, సమీక్ష సజావుగా సాగేందుకు ఈ స్టెప్స్‌ను పూర్తి చేయండి.

1. భాషను ఫాంట్ మ్యాపింగ్‌కు జోడించండి
   - `src/co_op_translator/fonts/font_language_mappings.yml` ను ఎడిట్ చేయండి
   - ఈ వివరాలతో ఒక ఎంట్రీ జోడించండి:
     - `code`: ISO-లాంటిది భాష కోడ్ (ఉదా., `vi`)
     - `name`: Human-friendly డిస్ప్లే పేరు
     - `font`: స్క్రిప్ట్‌కు మద్దతు ఇచ్చే `src/co_op_translator/fonts/` లోని ఫాంట్
     - `rtl`: రైట్-టు-లెఫ్ట్ అయితే `true`, లేకపోతే `false`

2. అవసరమైన ఫాంట్ ఫైళ్లను చేర్చండి (అవసరమైతే)
   - కొత్త ఫాంట్ అవసరమైతే, ఓపెన్ సోర్స్ డిస్ట్రిబ్యూషన్ కోసం లైసెన్స్ అనుకూలతను ధృవీకరించండి
   - ఫాంట్ ఫైల్‌ను `src/co_op_translator/fonts/` లో చేర్చండి

3. స్థానిక ధృవీకరణ
   - చిన్న నమూనా కోసం అనువాదాలను రన్ చేయండి (Markdown, images, మరియు notebooks అవసరమైతే)
   - అవుట్‌పుట్ సరిగ్గా రెండర్ అవుతుందా, ఫాంట్లు మరియు RTL లేఅవుట్ సహా ధృవీకరించండి

4. డాక్యుమెంటేషన్‌ను అప్‌డేట్ చేయండి
   - భాష `getting_started/supported-languages.md` లో కనిపించడాన్ని నిర్ధారించండి
   - `getting_started/README_languages_template.md` లో మార్పులు అవసరం లేదు; ఇది మద్దతు జాబితా నుండి జనరేట్ చేయబడుతుంది

5. PR ఓపెన్ చేయండి
   - జోడించిన భాష మరియు ఏదైనా ఫాంట్/లైసెన్సింగ్ అంశాలను వివరించండి
   - సాధ్యమైతే రెండర్ అవుట్‌పుట్ స్క్రీన్‌షాట్లను జోడించండి

Example YAML entry:

```yaml
new_lang(code):
  name: "New Language"
  font: "NotoSans-Medium.ttf"
  rtl: false
```

### కొత్త భాషను టెస్ట్ చేయండి

కొత్త భాషను టెస్ట్ చేయడానికి ఈ కింది కమాండ్‌ను రన్ చేయండి:

```bash
# వర్చువల్ ఎన్విరాన్‌మెంట్ సృష్టించి, సక్రియం చేయండి
python -m venv .venv
# విండోస్
.venv\Scripts\activate
# మాక్‌ఓఎస్/లినక్స్

pip install -e .
```

## నిర్వహకులు

### Commit మెసేజ్ మరియు Merge స్ట్రాటజీ

మా ప్రాజెక్ట్ యొక్క commit history లో స్థిరత్వం మరియు స్పష్టతను నిర్ధారించడానికి, **Squash and Merge** స్ట్రాటజీ ఉపయోగించినప్పుడు **final commit message** కోసం ప్రత్యేక commit మెసేజ్ ఫార్మాట్‌ను మేము అనుసరిస్తాము.

Pull request (PR) merge అయినప్పుడు, వ్యక్తిగత commits ఒక commit గా squash చేయబడతాయి. ప్రాజెక్ట్ యొక్క చరిత్రను శుభ్రంగా మరియు స్థిరంగా ఉంచడానికి ఫైనల్ commit మెసేజ్ క్రింది ఫార్మాట్‌ను అనుసరించాలి.

#### Commit మెసేజ్ ఫార్మాట్ (Squash and Merge కోసం)

మేము commit మెసేజ్‌ల కోసం ఈ ఫార్మాట్‌ను ఉపయోగిస్తాము:

```bash
<type>: <description> (#<PR సంఖ్య>)
```

- **type**: commit యొక్క వర్గాన్ని నిర్దేశిస్తుంది. మేము ఈ కేటగిరీలను ఉపయోగిస్తాము:
  - `Docs`: డాక్యుమెంటేషన్ అప్‌డేట్స్ కోసం.
  - `Build`: బిల్డ్ సిస్టమ్ లేదా డిపెండెన్సీలకు సంబంధించిన మార్పుల కోసం, కాన్ఫిగరేషన్ ఫైళ్లు, CI వర్క్‌ఫ్లోలు లేదా Dockerfile అప్‌డేట్స్ సహా.
  - `Core`: ప్రాజెక్ట్ యొక్క కోర్ ఫంక్షనాలిటీ లేదా ఫీచర్లలో మార్పుల కోసం, ముఖ్యంగా `src/co_op_translator/core` డైరెక్టరీలోని ఫైళ్లకు సంబంధించినవి.

- **description**: మార్పు యొక్క సంక్షిప్త సారాంశం.
- **PR number**: commit కు సంబంధించిన pull request సంఖ్య.

**Examples**:

- `Docs: Update installation instructions for clarity (#50)`
- `Core: Improve handling of image translation (#60)`

> [!NOTE]
> ప్రస్తుతం, **`Docs`**, **`Core`**, మరియు **`Build`** prefixes PR టైటిల్స్‌కు మార్పు చేసిన సోర్స్ కోడ్‌కు వర్తించే లేబుల్స్ ఆధారంగా స్వయంచాలకంగా జోడించబడతాయి. సరైన లేబుల్ వర్తించబడిందని నిర్ధారించండి, మీరు సాధారణంగా PR టైటిల్‌ను మాన్యువల్‌గా అప్‌డేట్ చేయాల్సిన అవసరం లేదు. మీరు అన్ని సరిగ్గా ఉన్నాయని మరియు prefix సరైన విధంగా జనరేట్ చేయబడిందని ధృవీకరించాలి.

#### Merge స్ట్రాటజీ

మేము pull requests కోసం **Squash and Merge** ను మా డిఫాల్ట్ స్ట్రాటజీగా ఉపయోగిస్తాము. ఈ స్ట్రాటజీ commit మెసేజ్‌లు మా ఫార్మాట్‌ను అనుసరించడాన్ని నిర్ధారిస్తుంది, వ్యక్తిగత commits చేయకపోయినా.

**కారణాలు**:

- శుభ్రమైన, లీనియర్ ప్రాజెక్ట్ చరిత్ర.
- commit మెసేజ్‌లలో స్థిరత్వం.
- చిన్న commits నుండి శబ్దం తగ్గింపు (ఉదా., "fix typo").

Merge చేసే సమయంలో, ఫైనల్ commit మెసేజ్ commit మెసేజ్ ఫార్మాట్‌ను అనుసరిస్తుందా అని నిర్ధారించండి.

**Squash and Merge ఉదాహరణ**
ఒక PR ఈ commits కలిగి ఉంటే:

- `fix typo`
- `update README`
- `adjust formatting`

ఇవి squash చేయబడాలి:
`Docs: Improve documentation clarity and formatting (#65)`
